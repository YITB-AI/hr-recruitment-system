Switch tenant resolution from subdomain to session-derived

The subdomain-based login deployed in the previous commit doesn't
actually work: dax-hr.vercel.app is a Vercel-provided *.vercel.app
alias, not a custom domain, so Vercel can't issue a wildcard TLS
certificate for its subdomains. Confirmed with a real HTTPS request
against the deployed subdomain (TLS handshake failure), not just a
DNS lookup.

Switched to session-derived resolution instead: the login form now
has a "Company ID" field (the company's slug), and companyId lives
entirely in the session from then on - no more Host-header parsing.

- proxy.ts: removed subdomain parsing, back to a plain cookie-presence
  check.
- lib/auth/session.ts: verifySessionToken no longer takes a tenantSlug
  parameter - the session token itself is the sole source of truth for
  companyId.
- Removed ROOT_DOMAIN/DEV_TENANT_SLUG everywhere (env schema,
  .env.local, Vercel production env vars) - genuinely dead now.
- Login form gained a Company ID field; loginAction resolves the
  company from it instead of the request's Host header.
- getAppearanceStyle() (root layout, renders pre-login too) switched
  from subdomain resolution to verifySession() (never redirects),
  falling back to the default style when logged out.
- scripts/create-company.ts now prints the Company ID to hand to the
  new admin instead of a subdomain login URL.

Verified: full build clean, session-verification suite re-run against
the new signature (7/7, including confirming a session token is only
ever resolvable to the one company it was created for), new login form
confirmed rendering the Company ID field over real HTTP.
